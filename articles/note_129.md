# 【AI対話アプリ】200の問い #129「エンジニア・バグの哲学」

このアプリは「エンジニア・バグの哲学」のためのAI対話プロンプト集です。コード上の「バグ」をただの敵ではなく「改善の贈り物」として再解釈し、デバッグのプロセスを通じて「問題解決の本質」を学ぶツールです。

---


## 1. 「バグ」を「設計の欠陥告発」と捉え直す

バグは、単なる「実装ミス」ではなく「設計や要件定義の欠陥」を教えてくれます。ユーザーが50%失敗するということは「あなたが想定していなかった使い方」か「あなたが見落とした条件」が存在するということです。AIに「このバグが示している『設計の欠陥』は何か。なぜそれを実装時に気づけなかったのか」と聞きます。バグは敵ではなく「要件定義を厳密にする機会」です。


## 2. 「バグを再現するプロセス」を最大限に詳しく分析する

バグは「なぜ起きたのか」の理由を隠しています。その理由を引き出すまでのプロセスが、エンジニアとしての成長を決めます。AIに「このバグが『どの入力値』『どの実行パス』で発生するのか。その条件を『ユーザーがそうしたくなる理由』まで含めて200字で分析して」と聞きます。バグは「起きた現象」ではなく「起きるに至った物語」として読み直すことで、同じようなバグを次の実装で予防できます。


## 3. 「バグの根本原因」まで掘り下げる。表面の原因で止まらない

バグの「直し方」は誰でもできます。大切なのは「なぜそこで間違えたのか」です。AIに「このバグの『表面の原因』『実装上の原因』『設計上の原因』『思考上の原因』の4段階で分析して」と聞きます。多くの場合、表面の原因（「配列の添字を間違えた」）の背景には、思考上の原因（「このパターンは発生しないだろうと仮定した」）が隠れています。


## 4. 「バグを修正した後の検証」を最大化する

バグを直すことは終わりではなく、始まりです。AIに「このバグが再発しないために、あなたが追加すべき『テストケース』は何か。ユーザーが『こんなことをしたい』という10のシナリオを、バグから逆算して設計してください」と聞きます。バグは「テストの漏れ」を教えてくれる羅針盤です。


## 体験談: 「バグ修正を『徹夜の苦労』から『設計の学習』に転換させた」

正直に書きます。最初、私はバグを「自分の実装ミス」として罪悪感を感じていました。責任者に報告書を出す時も「申し訳ありません」という語気でした。ただし、その夜の修正作業中にAIに相談しました。「このバグが示している『設計の欠陥』は何か」という問いです。分析を深めていく中で、気づいたのは「要件定義の段階で『こういう使い方がある』という想定が、実は営業ドキュメントには書いてあったのに、エンジニアには伝わっていなかった」ということです。バグは「実装ミス」ではなく「コミュニケーション欠陥」だったのです。その気づきから、3時間で修正が完了し、翌日の責任者への報告も「申し訳ありません」ではなく「設計・要件定義段階で、営業チームとエンジニアチームの認識にズレがありました。次の実装では、こういった認識合わせのプロセスを追加します」と建設的な提案になりました。バグは『敵』ではなく『成長の機会』だったのです。

- [ ] バグが「起きるに至った物語」まで含めて再現プロセスを分析した

- [ ] バグが再発しないための「テストケース10項目」を設計した


## 🤔 5つの問い (このアプリの中核)

1. このバグが生まれた直接的な原因は何ですか（添字、条件分岐、データ型など）
2. その原因が生まれた背景には、あなたのどんな『仮定』がありましたか
3. その『仮定』が間違っていたことを、今あなたは何から学びましたか
4. このバグが『設計段階で気づかれなかった』のは、どこのコミュニケーションが欠けていたからですか
5. 今後、同じ『種類のバグ』を予防するために、あなたが追加すべきテストケースは何ですか
